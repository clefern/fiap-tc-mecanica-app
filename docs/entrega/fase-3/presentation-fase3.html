Autenticação Serverless — Fase 3
Contrato do JWT e Cobertura de Testes
Estrutura do token emitido pela Lambda · Justificativa da Opção A do ADR-032 · Cenários cobertos
Contrato do JWT da Lambda
CampoValor
AlgorithmHS256
SecretSECURITY_JWT_SECRET_KEY (mesma do K8s Secret do app)
subjectemail do cliente
idUUID do cliente
role"CLIENTE"
iat / expEmissão + TTL 3600s
Por que Opção A (app não muda)
  • Zero novo controller no app — JwtAuthenticationFilter já valida transparentemente
  • Cliente faz 1 hop até Lambda direto (sem passar pelo app)
  • Sem ponto de falha extra no app (timeout/retry/HTTP client)
  • Encaixa no IngressRoute Traefik que já estava definido
  • Alternativas B (proxy), C (inline app — fere requisito serverless), D (Cognito RS256) descartadas e justificadas no ADR-032
Cobertura de Testes
  • JwtFromExternalIssuerIT (novo, app/integration) — 3 cenários: (1) JWT externo válido com subject = email_cadastrado é aceito e libera endpoint com object-level security; (2) JWT com subject inexistente → 401; (3) JWT com secret diferente → 401
  • Testes unitários de JwtService e JwtAuthenticationFilter mantidos (Fase 1)
  • Testes da Lambda (tests/cpf.test.ts, tests/handler.test.ts) — validação CPF módulo 11, mock de repository
  • Smoke E2E real (com Lambda deployada): roda após Provision & Deploy em lab

  • Receivers: otlp (gRPC 4317 + HTTP 4318), hostmetrics (root_path /hostfs, 60s), kubeletstats
  • Processors: batch (10s timeout)
  • Exporters: debug + otlphttp/newrelic (https://otlp.nr-data.net)
  • Pipelines: traces, metrics, logs (3 canais ativos)
  • RBAC: ServiceAccount + ClusterRole pra kubeletstats (get nodes/pods/services)
Instrumentação no App
  • OTel Java Agent v2.26.1 no Dockerfile (-javaagent:/app/otel.jar)
  • Micrometer OTLP exporter (management.otlp.metrics)
  • Logback JSON com LogstashLogbackEncoder (profile !local)
  • MonitoredOperationAspect instrumenta 14+ métodos com tags status/operation/service/error_type
  • CorrelationIdFilter propaga X-Correlation-ID via MDC entre logs e traces
  • Auditoria de OS: 3 eventos novos + OsHistoryListener + tabela os_history (migration V17)
Dashboards New Relic (como código em 17-observability.tf)

📊 Visão Geral de Negócio

  • Volume diário de OS criadas
  • Operações por tipo (OS / Orçamento)
  • Latência das APIs (p50/p95/p99)
  • Falhas de processamento por status
  • Ciclo de vida da OS (tempo médio por status)

⚙️ Infraestrutura (OTel / Kubernetes)

  • CPU / memória por pod e nó
  • Container metrics (restarts, OOM)
  • JVM heap, GC, threads
  • Throughput de requisições por endpoint
  • Conexões ao RDS (pool ativo / idle)
Alertas NRQL (3 condições críticas — IaC)
NomeCondiçãoAção
high_error_rateTaxa de erro > 5% por 300sNotifica canal de incidentes — pode indicar problema sistêmico (deploy ruim, RDS lento, Lambda quebrada)
os_process_failureErros em metricName LIKE 'os_%' (60s, at_least_once)Falha específica no processamento de OS — dispara investigação no log com correlationId
app_downtimeLoss of signal > 5minApp parou de enviar telemetria — pode indicar crash, OOM, network partition. Acionamento de plantão
Portabilidade garantida pelo OTel: trocar New Relic por Datadog, Honeycomb, Tempo ou self-hosted exige apenas mudar o exporter no otel-collector-configmap.yaml — zero alteração no código do app. Queries NRQL dos dashboards são o único lock-in.
