Sobre o Projeto

Este projeto foi desenvolvido como entrega do Tech Challenge da Pós-Graduação em Arquitetura de Software da FIAP. O objetivo é demonstrar a aplicação prática de padrões avançados de arquitetura de software — Hexagonal Architecture, Domain-Driven Design, Event-Driven patterns — em um sistema real de gestão de oficinas mecânicas, com infraestrutura cloud-native na AWS.

O Produto
Visão Geral do Projeto
Problema de negócio, solução técnica e stack adotada
O Problema
  • Oficinas mecânicas operam com processos manuais e planilhas
  • Ausência de rastreabilidade nas ordens de serviço
  • Aprovação de orçamentos por ligação ou presencialmente
  • Sem controle de estoque integrado ao fluxo de atendimento
  • Dificuldade em gerar relatórios e métricas operacionais
  • Problemas de segurança no acesso às informações de clientes
A Solução
  • Backend completo com API REST documentada via Swagger
  • Máquina de estados para gestão do ciclo de vida das OS
  • Aprovação de orçamentos via link seguro (HMAC-SHA256)
  • Controle de estoque com reserva automática na aprovação
  • Autenticação JWT com controle de acesso por perfil
  • Infraestrutura cloud-native escalável na AWS com EKS
Stack Tecnológica

Backend

  • Java 21 (LTS) + Records & Sealed Classes
  • Spring Boot 3.3.4 + Spring Security 6
  • Spring Data JPA + Hibernate 6.x
  • JJWT 0.12.5 — tokens JWT seguros
  • Bucket4j — rate limiting
  • Flying Saucer + OpenPDF — geração PDF

Dados & Qualidade

  • PostgreSQL 16 — banco de dados principal
  • Flyway — migrations versionadas (V1–V16)
  • H2 In-Memory — testes unitários
  • TestContainers — testes integrados
  • JaCoCo — cobertura 90% / 80% domínio
  • SonarQube — análise de qualidade

Infra & DevOps

  • Docker multi-stage + distroless
  • Kubernetes (EKS 1.33) + HPA
  • Terraform IaC — VPC, EKS, ECR, ALB
  • GitHub Actions — CI/CD completo
  • Amazon ECR — registry de imagens
  • NGINX Ingress + Cert-Manager
Escopo: API backend RESTful com 13 domínios funcionais, autenticação e autorização completa, domain events, geração de PDF, envio de email, observabilidade (Actuator + métricas) e infraestrutura AWS provisionada por Terraform — tudo com cobertura de testes abrangente.

Máquina de Estados — Ordem de Serviço
RECEBIDA → EM_DIAGNOSTICO → AG.APROVACAO
APROVADA → EM_EXECUCAO → FINALIZADA → ENTREGUE
CANCELADA ← qualquer estado

Transições inválidas lançam TransicaoStatusInvalidaException (422).

Regra de Dependência
  • Setas sempre apontam para o domínio (nunca o contrário)
  • Domain layer: zero anotações Spring, zero JPA — POJO puro
  • Ports: interfaces Java no domínio, implementações na infra
  • Adapters JPA implementam as interfaces de repositório do domínio
  • Dois mappers: infra/mapper/ e presentation/mapper/

POST /api/integracoes/orcamentos/aprovacao API Key M2M Aprovação de orçamento por integração máquina-a-máquina
Email com Link Seguro (HMAC-SHA256)
  • Ao gerar orçamento, email automático é enviado ao cliente
  • Link único com token HMAC-SHA256 assinado com segredo
  • Token contém ID do orçamento + timestamp de expiração
  • Validade configurável (padrão: 48h)
  • Endpoint público para aprovação: nenhuma senha necessária
  • Implementado via HmacUtils + Thymeleaf template
Docker Multi-Stage Build
  • Dockerfile — multi-stage com JRE Alpine (imagem mínima ~120MB)
  • Dockerfile.deploy — distroless para produção (zero shell)
  • Stage 1: Maven build com cache de dependências
  • Stage 2: apenas JRE + artefato compilado
  • Usuário não-root em todas as imagens (segurança)
  • Trivy scan integrado no pipeline de build

Workflows
Workflow Trigger Arquivo Responsabilidade
CI Push / PR → develop ci.yml Compilação Maven, cache M2, lint (Checkstyle/PMD/SpotBugs)
Test Push / PR → develop test.yml Testes unitários e integrados (TestContainers), cobertura JaCoCo
Build Merge → main build.yml Docker multi-stage + Trivy scan + push para ECR
CD Após build cd.yml Deploy em EKS via kubectl apply + k8s-deploy.sh (envsubst)
Infra Manual dispatch infra.yml Terraform plan/apply — provisionamento da infraestrutura AWS
Segurança no Pipeline
  • Secrets via GitHub Actions Secrets (nunca expostos em logs)
  • OIDC para autenticação AWS — sem chaves de longa duração
  • Trivy: scan de vulnerabilidades na imagem Docker
  • EBS CSI Driver com OIDC para volumes persistentes
  • Token Sonar removido do código (ADR-SEC-001)
Qualidade no Pipeline
  • Cache Maven: build ~40% mais rápido
  • JaCoCo: falha se cobertura < 90% global
  • SonarQube analysis em cada merge
  • Pre-commit hook: Spotless auto-format
  • 3 índices PostgreSQL em V16 (status, data)

ADRs de Maior Impacto ⭐
ADR-010 — Autorização Híbrida

JWT roles + segurança a nível de objeto (OsSecurity) — clientes só acessam suas próprias OS

ADR-019 — Estoque Desacoplado

Estoque deduzido na aprovação do orçamento, não do budget — evita inconsistências

ADR-022 — Fluxo Invertido OS/Orçamento

Cliente aprova orçamento → OS avança para APROVADA (não o contrário)

ADR-031 — Decisões Fase 2

PostgreSQL em PVC vs RDS, OIDC para EBS, pipeline separado CI vs CD
